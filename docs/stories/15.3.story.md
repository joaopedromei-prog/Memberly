## Story

**Como** membro da plataforma,
**Quero** receber notificações automáticas quando uma nova aula é publicada, alguém responde ao meu comentário, ou completo um curso,
**Para que** eu fique informado sobre atualizações relevantes sem precisar verificar manualmente.

## Acceptance Criteria

1. **AC1** — Serviço `src/lib/notifications/create-notification.ts` exporta função `createNotification(params: CreateNotificationParams): Promise<void>` que insere uma notificação na tabela `notifications` usando o admin client (service role, bypass RLS).
2. **AC2** — O serviço verifica `notification_preferences` antes de criar: se o membro desabilitou o tipo de notificação, não cria. Se não há registro de preferência, assume `enabled = true` (default).
3. **AC3** — Tipo `NEW_LESSON`: Quando admin publica uma lesson (PATCH `/api/lessons/[id]` com `is_published: true`), notifica todos os membros com `member_access` ao produto da lesson. Título: "Nova aula disponível". Body: "{lessonTitle} em {productTitle}". Data: `{ lessonId, productSlug, moduleId }`.
4. **AC4** — Tipo `COMMENT_REPLY`: Quando alguém cria um comentário com `parent_id` (POST `/api/lessons/[id]/comments`), notifica o autor do comentário pai. Não notifica se o autor da resposta é o mesmo do comentário pai. Título: "Nova resposta ao seu comentário". Body: "{replierName} respondeu ao seu comentário em {lessonTitle}". Data: `{ lessonId, commentId, parentCommentId }`.
5. **AC5** — Tipo `COURSE_COMPLETED`: Quando `lesson_progress` marca completed=true (POST `/api/progress/[lessonId]`), verificar se o membro completou 100% das aulas publicadas do produto. Se sim, notificar o membro. Título: "Curso concluído!". Body: "Parabéns! Você completou {productTitle}". Data: `{ productId, productSlug }`.
6. **AC6** — As notificações são criadas de forma assíncrona (fire-and-forget) — falha na criação de notificação NÃO deve impedir a operação principal (publish lesson, create comment, mark progress).
7. **AC7** — Constantes de tipo exportadas: `NOTIFICATION_TYPES = { NEW_LESSON: 'NEW_LESSON', COMMENT_REPLY: 'COMMENT_REPLY', COURSE_COMPLETED: 'COURSE_COMPLETED' } as const`.
8. **AC8** — Testes unitários para o serviço de criação e para cada trigger integration point.

### Core Service Pattern

```typescript
// src/lib/notifications/create-notification.ts
import { createAdminClient } from '@/lib/supabase/admin';
import { NOTIFICATION_TYPES } from './types';

interface CreateNotificationParams {
  profileId: string;
  type: keyof typeof NOTIFICATION_TYPES;
  title: string;
  body: string;
  data?: Record<string, unknown>;
}

export async function createNotification(params: CreateNotificationParams): Promise<void> {
  const adminClient = createAdminClient();

  // Check user preference
  const { data: pref } = await adminClient
    .from('notification_preferences')
    .select('enabled')
    .eq('profile_id', params.profileId)
    .eq('notification_type', params.type)
    .maybeSingle();

  // If preference exists and is disabled, skip
  if (pref && !pref.enabled) return;

  await adminClient.from('notifications').insert({
    profile_id: params.profileId,
    type: params.type,
    title: params.title,
    body: params.body,
    data: params.data ?? null,
  });
}
```

### NEW_LESSON Trigger Pattern

```typescript
// src/lib/notifications/triggers/new-lesson.ts
import { createAdminClient } from '@/lib/supabase/admin';
import { createNotification } from '../create-notification';
import { NOTIFICATION_TYPES } from '../types';

export async function notifyNewLesson(lessonId: string): Promise<void> {
  const adminClient = createAdminClient();

  // Get lesson with module and product info
  const { data: lesson } = await adminClient
    .from('lessons')
    .select('title, module:modules!inner(product_id, product:products!inner(title, slug))')
    .eq('id', lessonId)
    .single();

  if (!lesson) return;

  const module = lesson.module as unknown as { product_id: string; product: { title: string; slug: string } };

  // Get all members with access to this product
  const { data: members } = await adminClient
    .from('member_access')
    .select('profile_id')
    .eq('product_id', module.product_id);

  if (!members?.length) return;

  // Create notifications for all members (fire-and-forget per member)
  await Promise.allSettled(
    members.map((m) =>
      createNotification({
        profileId: m.profile_id,
        type: 'NEW_LESSON',
        title: 'Nova aula disponível',
        body: `${lesson.title} em ${module.product.title}`,
        data: { lessonId, productSlug: module.product.slug, moduleId: module.product_id },
      })
    )
  );
}
```

### NEW_LESSON Publish Detection

No endpoint `PATCH /api/lessons/[id]`, para detectar que a lesson está sendo publicada pela primeira vez, buscar o estado anterior antes do update:

```typescript
// Before update
const { data: currentLesson } = await supabase
  .from('lessons')
  .select('is_published')
  .eq('id', lessonId)
  .single();

// After successful update
if (body.is_published === true && currentLesson && !currentLesson.is_published) {
  notifyNewLesson(lessonId).catch(() => {/* silent */});
}
```

### Edge Cases

- **Self-reply:** Não notificar quando o autor da resposta é o mesmo do comentário pai (AC4).
- **Admin publica e despublica:** Só notificar na transição `false → true`, não em updates subsequentes onde `is_published` já é `true`.
- **Batch notifications NEW_LESSON:** Usar `Promise.allSettled()` para que falha em um membro não impeça notificação dos outros.
- **Course completion idempotente:** Verificar se já existe notificação COURSE_COMPLETED para o par (profile_id, product) antes de criar duplicata. Ou aceitar duplicatas — membro pode completar e descomplete e completar novamente.
- **Performance:** Para produtos com muitos membros, considerar batch insert em vez de individual `createNotification()` calls. Para v1, individual é aceitável.
