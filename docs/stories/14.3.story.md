## Story

**Como** membro que completou 100% das aulas de um produto,
**Quero** ver um botão de download de certificado na página do produto,
**Para que** eu possa gerar e baixar meu certificado de conclusão diretamente.

## Acceptance Criteria

1. **AC1** — Na página do produto (`/products/[slug]`), quando o membro completou 100% das aulas publicadas, exibe um botão "Baixar Certificado" com ícone de download.
2. **AC2** — Quando o progresso é inferior a 100%, o botão não aparece. Em vez disso, exibe a barra de progresso existente com percentual (ex: "72% concluído").
3. **AC3** — Ao clicar no botão, o sistema chama `POST /api/certificates/generate` (Story 14.2). Se o certificado já existe, retorna o existente.
4. **AC4** — Durante a geração/download, o botão exibe estado de loading (spinner + "Gerando certificado...").
5. **AC5** — Após receber a URL do certificado, o sistema inicia o download automático do PDF.
6. **AC6** — Se a geração falhar, exibe toast de erro com mensagem amigável.
7. **AC7** — O botão segue o design system Netflix-style: fundo `--color-primary` (#e50914), texto branco, ícone de certificado/download.
8. **AC8** — O componente verifica se o produto tem `certificate_enabled = true` (Story 14.4) antes de exibir o botão. Se certificados estão desabilitados para o produto, o botão não aparece mesmo com 100% concluído.
9. **AC9** — Em modo preview (admin), o botão aparece com label "(Preview)" e não gera certificado real.

### Download Pattern

```typescript
const handleDownload = async () => {
  setLoading(true);
  try {
    const res = await fetch('/api/certificates/generate', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ productId }),
    });
    const data = await res.json();
    if (!res.ok) throw new Error(data.error?.message);

    // Trigger download
    window.open(data.certificate_url, '_blank');
  } catch (err) {
    addToast({ type: 'error', message: 'Erro ao gerar certificado. Tente novamente.' });
  } finally {
    setLoading(false);
  }
};
```

### Dependency on Story 14.4

O campo `certificate_enabled` na tabela `products` será criado na Story 14.4. Se 14.3 for implementada antes de 14.4, usar fallback `true` (certificados habilitados por padrão) e adicionar a verificação do campo quando 14.4 estiver pronta.

### Edge Cases

- **Produto sem aulas publicadas:** `totalLessons === 0` — botão não aparece.
- **API retorna certificado existente:** Mesmo fluxo — download da URL retornada.
- **PDF ainda não está no Storage:** A API gera e faz upload antes de retornar a URL.
- **Mobile:** O botão deve ser responsivo. Em mobile, `window.open` abre nova aba com o PDF.

### Testing

- Render com `isComplete=false` → botão não renderiza
- Render com `isComplete=true, certificateEnabled=true` → botão visível
- Render com `isComplete=true, certificateEnabled=false` → botão não renderiza
- Click → loading state → success → window.open chamado
- Click → loading state → error → toast exibido
- Render com `isPreview=true` → botão mostra "(Preview)"

## QA Results

### Gate Decision: FAIL
**Reviewed by:** @qa (Quinn) — 2026-03-20
**AC Coverage:** 7/9 (78%)

- **AC1** — PASS: Button renders with download icon when `isComplete=true` and `certificateEnabled=true`. Does not render when incomplete.
- **AC2** — PASS: Button returns `null` when `isComplete=false`. Progress bar is handled by existing `ProductHero` component.
- **AC3** — FAIL (BUG): The component calls `POST /api/certificates/generate` correctly, but reads the response incorrectly. Line 52: `window.open(data.certificate_url, '_blank')` accesses `data.certificate_url`, but the actual API response from Story 14.2 is `{ certificate: { id, hash, certificate_url, issued_at } }` — the URL is at `data.certificate.certificate_url`. This would result in `window.open(undefined, '_blank')`. The unit test masks this bug by mocking the response as `{ certificate_url: '...' }` (flat) instead of the real nested structure.
- **AC4** — PASS: Loading state shows spinner + "Gerando certificado..." text. Button is disabled during loading.
- **AC5** — CONDITIONAL FAIL: Download via `window.open` would fail due to the AC3 bug above (undefined URL).
- **AC6** — PASS: Error toast calls `addToast('Erro ao gerar certificado. Tente novamente.', 'error')`. Handles both API errors and network failures.
- **AC7** — PASS: Button uses `bg-primary`, white text, hover effects consistent with design system.
- **AC8** — PASS: Component checks `certificateEnabled` prop and returns `null` when false. Product page passes `certificate_enabled` from DB query.
- **AC9** — PASS: Preview mode shows "(Preview)" label and button is disabled (no API call).

**Issues:**

1. **[BLOCKING] Response field path mismatch:** `CertificateDownloadButton.tsx` line 52 reads `data.certificate_url` but the API returns `data.certificate.certificate_url`. Fix: change to `window.open(data.certificate.certificate_url, '_blank')`.
2. **[BLOCKING] Test mock does not match real API response:** `CertificateDownloadButton.test.tsx` line 101 mocks response as `{ certificate_url: certificateUrl }` instead of `{ certificate: { certificate_url: certificateUrl } }`. This masks the bug above.

**Observações:**
- The page integration in `products/[slug]/page.tsx` is well done — correct props, conditional rendering, layout consistency.
- 13 unit tests are thorough in coverage of states, but the mock for the success case does not match the real API contract.
- Fix is a 1-line change in the component + 2-line fix in the test. After fix, re-run tests and submit for re-review.
