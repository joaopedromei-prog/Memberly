## Story

**As a** admin,
**I want** visualizar, buscar e gerenciar membros e seus acessos,
**so that** eu tenha controle sobre quem acessa cada produto.

---

## Acceptance Criteria

1. Página `/admin/members` lista membros com: nome, email, data de registro, número de produtos com acesso
2. Busca por nome ou email com resultado em tempo real (debounced)
3. Filtro por produto (ver apenas membros com acesso a um produto específico)
4. Ao clicar em um membro, exibir detalhes: perfil, lista de produtos com acesso (com data de liberação e origem: webhook/manual)
5. Ação de atribuir acesso manual: selecionar produto e confirmar
6. Ação de remover acesso: selecionar produto e confirmar com warning
7. Paginação na lista de membros (preparado para 10K+ registros)

### Database Schema [Source: architecture.md#8-database-schema]

**profiles:**
```sql
CREATE TABLE profiles (
    id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
    full_name TEXT NOT NULL DEFAULT '',
    avatar_url TEXT,
    role TEXT NOT NULL DEFAULT 'member' CHECK (role IN ('member', 'admin')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**member_access:**
```sql
CREATE TABLE member_access (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    profile_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
    product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
    granted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    granted_by TEXT NOT NULL DEFAULT 'manual' CHECK (granted_by IN ('webhook', 'manual')),
    transaction_id TEXT,
    UNIQUE(profile_id, product_id)
);

CREATE INDEX idx_member_access_profile ON member_access(profile_id);
CREATE INDEX idx_member_access_product ON member_access(product_id);
CREATE INDEX idx_member_access_transaction ON member_access(transaction_id);
```

### Supabase Query Patterns

**List members with pagination, search, and product filter:**
```typescript
const supabase = await createServerSupabaseClient();

let query = supabase
  .from('profiles')
  .select('*, member_access(count)', { count: 'exact' })
  .eq('role', 'member')
  .order('created_at', { ascending: false });

// Search filter
if (search) {
  query = query.ilike('full_name', `%${search}%`);
}

// Product filter (join with member_access)
if (productId) {
  query = query.in(
    'id',
    supabase
      .from('member_access')
      .select('profile_id')
      .eq('product_id', productId)
  );
}

// Pagination
const from = (page - 1) * limit;
const to = from + limit - 1;
query = query.range(from, to);

const { data, error, count } = await query;
```

**NOTE on email search:** Supabase `profiles` table does NOT have an email column (email lives in `auth.users`). To search by email, you need to either:
1. Use `supabase.auth.admin.listUsers()` with service role key to search auth.users, then match profile IDs
2. Add an `email` column to `profiles` (denormalized, synced from auth.users)
3. Use a database view that joins profiles with auth.users

**Recommended approach:** Create a Postgres view or use `supabase.auth.admin.listUsers()` for email search. The simpler approach for MVP is option 1.

**Member detail with access list:**
```typescript
// Get profile
const { data: profile } = await supabase
  .from('profiles')
  .select('*')
  .eq('id', memberId)
  .single();

// Get email from auth (requires service role)
const { data: { user } } = await supabase.auth.admin.getUserById(memberId);
const email = user?.email;

// Get access list with product details
const { data: access } = await supabase
  .from('member_access')
  .select('*, products(id, title, slug, banner_url)')
  .eq('profile_id', memberId)
  .order('granted_at', { ascending: false });
```

**Grant access (manual):**
```typescript
const { data, error } = await supabase
  .from('member_access')
  .insert({
    profile_id: memberId,
    product_id: productId,
    granted_by: 'manual',
  })
  .select()
  .single();

// Handle unique constraint violation (already has access)
if (error?.code === '23505') {
  return apiError('CONFLICT', 'Member already has access to this product', 409);
}
```

**Revoke access:**
```typescript
const { error } = await supabase
  .from('member_access')
  .delete()
  .eq('profile_id', memberId)
  .eq('product_id', productId);
```

### Route Handler Template — Members List

```typescript
// src/app/api/members/route.ts
import { createServerSupabaseClient } from '@/lib/supabase/server';
import { apiError, apiSuccess } from '@/lib/utils/api-response';
import { type NextRequest } from 'next/server';

export async function GET(request: NextRequest) {
  const supabase = await createServerSupabaseClient();
  const { searchParams } = new URL(request.url);

  const page = Math.max(1, parseInt(searchParams.get('page') || '1'));
  const limit = Math.min(100, Math.max(1, parseInt(searchParams.get('limit') || '20')));
  const search = searchParams.get('search') || '';
  const productId = searchParams.get('product_id') || '';

  const from = (page - 1) * limit;
  const to = from + limit - 1;

  let query = supabase
    .from('profiles')
    .select('*, member_access(count)', { count: 'exact' })
    .eq('role', 'member')
    .order('created_at', { ascending: false })
    .range(from, to);

  if (search) {
    query = query.ilike('full_name', `%${search}%`);
  }

  const { data, error, count } = await query;

  if (error) return apiError('FETCH_ERROR', error.message, 500);

  return apiSuccess({
    data,
    total: count || 0,
    page,
    limit,
    totalPages: Math.ceil((count || 0) / limit),
  });
}
```

### Important Design Considerations

1. **Performance for 10K+ members:** Use server-side pagination (range queries), not client-side. Ensure DB indexes on `profiles.role` and `member_access.profile_id` are leveraged.

2. **Email access:** Since email is in `auth.users` and not `profiles`, the member detail endpoint needs to use `supabase.auth.admin.getUserById()` with the service role key. This is server-side only.

3. **Idempotency on grant access:** The `UNIQUE(profile_id, product_id)` constraint on `member_access` prevents duplicate grants. Handle the 23505 error code gracefully.

4. **Debounced search:** Implement debounce on the client side (300ms) to avoid excessive API calls. Use URL search params for shareable/bookmarkable filtered views.

### Coding Standards [Source: architecture.md#16-coding-standards]

- **Absolute Imports:** Always use `@/` prefix
- **Supabase Client:** `createServerSupabaseClient()` on server, `createClient()` on client
- **Error Handling:** All Route Handlers must return standardized `ApiError` format
- **RLS Trust:** Never duplicate access logic in code — trust Supabase RLS policies
- **Webhook Processing:** Use `supabase.auth.admin` (service role) for auth admin operations
