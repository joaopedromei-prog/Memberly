## Story

**As an** admin,
**I want** que operações de importação de membros e atualização de settings sejam rápidas e confiáveis,
**so that** importações grandes não deem timeout e atualizações de configuração sejam atômicas.

---

## Context

### Estado Atual

O Brownfield Assessment identificou 3 problemas de performance críticos:

1. **N+1 no member import** (`/api/members/import/route.ts`) — `admin.auth.admin.listUsers()` busca TODOS os usuários do Supabase Auth para verificar se um email já existe. Em bases com milhares de users, isso causa timeout.

2. **Sequential updates em settings** (`/api/settings/route.ts`) — Loop que faz 1 request por campo de configuração. Com 10 campos = 10 round-trips ao DB.

3. **Sequential updates em reorder** (`/api/products/[id]/modules/reorder/route.ts`) — Loop com N updates sequenciais para N módulos/lições.

### Mudança Necessária

1. **Member import**: Substituir `listUsers()` por lookup individual usando `admin.auth.admin.getUserByEmail(email)` com try/catch (retorna null se não encontrado)
2. **Settings**: Usar `upsert` com array de valores em vez de loop sequential
3. **Module reorder**: Usar uma única query batch com `Promise.all()` para updates paralelos (ou RPC function)

### Decisões Técnicas

- `getUserByEmail()` é O(1) vs `listUsers()` que é O(n) — melhoria direta
- Settings batch: usar `Promise.all()` para paralelizar upserts (Supabase não suporta multi-row upsert nativo via JS client para tabelas key-value)
- Module reorder: `Promise.all()` com updates individuais paralelos (aceitável para <100 módulos; RPC seria over-engineering)
