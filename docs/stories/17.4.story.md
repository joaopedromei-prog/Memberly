## Story

**Como** plataforma Memberly,
**Quero** persistir o numero de telefone do cliente recebido via webhook da Payt,
**Para que** o telefone fique disponivel no perfil do membro para futuros usos (login via SMS, contato, etc.).

## Acceptance Criteria

1. **AC1** — Migration adicionando coluna `phone` (TEXT, nullable) na tabela `profiles`.
2. **AC2** — `member-provisioning.ts` atualizado para salvar o `phone` do customer no `user_metadata` do Supabase Auth e na coluna `phone` do perfil.
3. **AC3** — Se o membro ja existe e nao tem telefone, o telefone do webhook atualiza o perfil (upsert).
4. **AC4** — Se o membro ja tem telefone, NAO sobrescrever com o do webhook (preservar dado existente).
5. **AC5** — Type `Profile` em `src/types/` atualizado para incluir `phone?: string`.
6. **AC6** — Admin pode ver o telefone do membro na listagem de membros.
7. **AC7** — Testes unitarios para logica de upsert do telefone (3 cases: sem telefone anterior, com telefone anterior, webhook sem telefone).

## Dev Notes

### Migration

```sql
ALTER TABLE profiles ADD COLUMN phone TEXT;
```

### Logica de upsert no member-provisioning.ts

```typescript
// Na criacao de novo membro:
user_metadata: {
  role: 'member',
  full_name: customerName,
  phone: customer.phone || null,
}

// Apos criar/encontrar membro, atualizar profile:
if (customer.phone && !existingProfile.phone) {
  await adminClient.from('profiles')
    .update({ phone: customer.phone })
    .eq('id', profileId);
}
```

### Payload Payt

O campo `customer.phone` ja e aceito pelo schema Zod (`paytCustomerSchema`) como `z.string().optional()`.
