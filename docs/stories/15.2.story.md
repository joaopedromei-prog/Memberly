## Story

**Como** membro autenticado,
**Quero** endpoints de API para listar, marcar como lida e deletar minhas notificações,
**Para que** o frontend possa exibir notificações, atualizar seu estado de leitura e obter contagem de não-lidas.

## Acceptance Criteria

1. **AC1** — `GET /api/notifications` retorna lista paginada de notificações do usuário autenticado, ordenada por `created_at` desc. Aceita query params `cursor` (string, created_at da última) e `limit` (number, default 10, max 50). Resposta: `{ notifications: [...], nextCursor: string | null }`.
2. **AC2** — `PATCH /api/notifications/[id]` marca uma notificação como lida, setando `read = true` e `read_at = NOW()`. Retorna a notificação atualizada. Retorna 404 se não encontrada ou não pertence ao usuário.
3. **AC3** — `POST /api/notifications/mark-all-read` marca todas as notificações não-lidas do usuário como lidas. Retorna `{ updated: number }` com a contagem de notificações atualizadas.
4. **AC4** — `DELETE /api/notifications/[id]` deleta uma notificação do usuário. Retorna `{ deleted: true }`. Retorna 404 se não encontrada ou não pertence ao usuário.
5. **AC5** — `GET /api/notifications/unread-count` retorna `{ count: number }` com o total de notificações não-lidas do usuário.
6. **AC6** — Todos os endpoints requerem autenticação (401 se não autenticado).
7. **AC7** — Todos os endpoints aplicam filtro por `profile_id = auth.uid()` — um membro não pode acessar notificações de outro.
8. **AC8** — Testes unitários para cada endpoint cobrindo happy path e edge cases.

### API Route Pattern (seguir `api/lessons/[id]/comments/route.ts`)

```typescript
import { authenticateUser } from '@/lib/utils/auth-guard';
import { apiError, apiSuccess } from '@/lib/utils/api-response';
import type { NextRequest } from 'next/server';

export async function GET(request: NextRequest) {
  const auth = await authenticateUser();
  if (!auth.ok) return auth.response;

  const { user, supabase } = auth.data;
  const { searchParams } = new URL(request.url);
  const cursor = searchParams.get('cursor');
  const limit = Math.min(Number(searchParams.get('limit')) || 10, 50);

  let query = supabase
    .from('notifications')
    .select('*')
    .eq('profile_id', user.id)
    .order('created_at', { ascending: false })
    .limit(limit);

  if (cursor) {
    query = query.lt('created_at', cursor);
  }

  const { data, error } = await query;
  if (error) return apiError('DATABASE_ERROR', 'Erro ao buscar notificações', 500);

  const hasMore = data.length === limit;
  const nextCursor = hasMore ? data[data.length - 1].created_at : null;

  return apiSuccess({ notifications: data, nextCursor });
}
```

const { data, error } = await supabase

### Mark All Read Pattern

```typescript
// POST /api/notifications/mark-all-read
const { count, error } = await supabase
  .from('notifications')
  .update({ read: true, read_at: new Date().toISOString() })
  .eq('profile_id', user.id)
  .eq('read', false)
  .select('id', { count: 'exact', head: true });

return apiSuccess({ updated: count ?? 0 });
```

### Unread Count Pattern

```typescript
// GET /api/notifications/unread-count
const { count, error } = await supabase
  .from('notifications')
  .select('id', { count: 'exact', head: true })
  .eq('profile_id', user.id)
  .eq('read', false);

return apiSuccess({ count: count ?? 0 });
```

### Edge Cases

- **Cursor pagination:** Use `created_at` como cursor (ISO string). Se duas notificações tiverem o mesmo `created_at`, a ordenação pode variar — aceitável para notificações.
- **Limit max 50:** Prevenir queries abusivas. Default 10.
- **Mark all read idempotente:** Se não houver notificações não-lidas, retorna `{ updated: 0 }` — não é erro.
- **Delete/PATCH com id inexistente:** Retornar 404, não 500.
- **Segurança:** Mesmo com RLS, usar filtro explícito `profile_id = user.id` em todas as queries como defesa em profundidade.

### Implementation Notes

- All endpoints follow established project patterns: `authenticateUser()` for auth, `apiSuccess()`/`apiError()` for responses, explicit `profile_id` filter for defense-in-depth alongside RLS.
- `GET /api/notifications` uses cursor-based pagination with `created_at` as cursor. Default limit 10, max 50, minimum 1.
- `PATCH /api/notifications/[id]` sets `read: true` and `read_at` to current timestamp. Idempotent (re-marking an already-read notification succeeds).
- `POST /api/notifications/mark-all-read` counts unread first, then updates. Returns `{ updated: 0 }` when no unread (idempotent).
- `DELETE /api/notifications/[id]` returns 404 for missing or other-user notifications.
- `GET /api/notifications/unread-count` returns `{ count: number }`.
- Fixed TypeScript issue: `.select()` after `.update()` in Supabase JS v2 does not accept `{ count, head }` options. Refactored mark-all-read to count first, then update.
- 24 integration tests covering happy paths, edge cases (empty lists, null counts, idempotent operations), 401 auth, 404 not found, and 500 database errors.
