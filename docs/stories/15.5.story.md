## Story

**Como** membro autenticado,
**Quero** uma página dedicada em `/notifications` com a lista completa de todas as minhas notificações,
**Para que** eu possa visualizar, filtrar por tipo, marcar como lidas e deletar notificações mais antigas.

## Acceptance Criteria

1. **AC1** — Página acessível em `(member)/notifications/page.tsx`, protegida por autenticação (middleware já cobre rotas member).
2. **AC2** — Lista paginada de todas as notificações do usuário (scroll infinito ou botão "Carregar mais"), usando `GET /api/notifications` com cursor pagination.
3. **AC3** — Filtro por tipo: tabs/pills com opções "Todas", "Novas aulas", "Respostas", "Concluídos". Filtra client-side nos dados carregados ou envia query param `type` para a API.
4. **AC4** — Cada notificação exibe: ícone por tipo, título, body completo (não truncado), data/hora formatada, indicador visual de lida/não-lida.
5. **AC5** — Ação "Marcar como lida" em cada notificação não-lida (click ou botão). Usa `PATCH /api/notifications/[id]`.
6. **AC6** — Botão "Marcar todas como lidas" no topo da página. Usa `POST /api/notifications/mark-all-read`.
7. **AC7** — Ação "Deletar" em cada notificação (ícone lixeira ou menu). Usa `DELETE /api/notifications/[id]`. Confirmação não necessária (ação reversível apenas via re-trigger).
8. **AC8** — Empty state quando não há notificações: ícone de sino, texto "Você não tem notificações" com estilo Netflix dark.
9. **AC9** — Empty state filtrado: "Nenhuma notificação deste tipo" quando filtro aplicado retorna vazio.
10. **AC10** — Clicar em uma notificação navega para a página relevante (mesma lógica de navegação do dropdown em 15.4).
11. **AC11** — Layout responsivo: cards em lista vertical, espaçamento consistente com design system dark theme.

### NotificationCard Component

```tsx
// src/components/member/NotificationCard.tsx
import { BookOpen, MessageCircle, Trophy, Trash2 } from 'lucide-react';
import { useRouter } from 'next/navigation';

const ICONS = { NEW_LESSON: BookOpen, COMMENT_REPLY: MessageCircle, COURSE_COMPLETED: Trophy };

interface Props {
  notification: NotificationType;
  onMarkRead: (id: string) => void;
  onDelete: (id: string) => void;
}

export function NotificationCard({ notification, onMarkRead, onDelete }: Props) {
  const router = useRouter();
  const Icon = ICONS[notification.type] ?? Bell;

  function handleClick() {
    if (!notification.read) onMarkRead(notification.id);
    const url = getNotificationUrl(notification);
    router.push(url);
  }

  return (
    <div
      onClick={handleClick}
      className={`flex items-start gap-4 p-4 rounded-lg cursor-pointer transition-colors hover:bg-dark-card ${
        !notification.read ? 'bg-dark-card/50 border-l-2 border-primary' : 'bg-transparent'
      }`}
    >
      <Icon className="text-neutral-400 mt-1 shrink-0" size={20} />
      <div className="flex-1 min-w-0">
        <p className="text-sm font-medium text-white">{notification.title}</p>
        <p className="text-sm text-neutral-400 mt-1">{notification.body}</p>
        <p className="text-xs text-neutral-500 mt-2">{formatDateTime(notification.created_at)}</p>
      </div>
      <button onClick={(e) => { e.stopPropagation(); onDelete(notification.id); }} ...>
        <Trash2 size={16} />
      </button>
    </div>
  );
}
```

### Design System

- Page background: inherits `bg-dark-bg` from member layout
- Card unread: `bg-dark-card/50 border-l-2 border-primary`
- Card read: `bg-transparent hover:bg-dark-card`
- Filter tab active: `bg-primary text-white`
- Filter tab inactive: `bg-dark-surface text-neutral-400 hover:text-white`
- "Carregar mais" button: `bg-dark-surface hover:bg-dark-card text-neutral-300`
- Empty state icon: `text-neutral-600` large Bell icon
- Empty state text: `text-neutral-500`

### Edge Cases

- **Client-side filter com pagination:** Se aplicar filtro e os dados carregados não contiverem itens daquele tipo, o usuário vê empty state mesmo que existam mais no servidor. Solução v1: carregar mais agressivamente (limit=50) ou adicionar query param `type` na API. Para v1, filtro client-side com limit=20 é aceitável.
- **Delete otimista:** Remover da lista local imediatamente, reverter se API falhar.
- **Mark all read otimista:** Atualizar todos localmente como lidos imediatamente.
- **Navegação de notificação:** `getNotificationUrl()` deve ser compartilhado com `NotificationDropdown` (15.4) — extrair para util.

#### Implementation Details

- **Pagination:** Cursor-based with "Carregar mais" button, limit=20, uses existing `GET /api/notifications` API
- **Filtering:** Client-side via 4 tabs (Todas, Novas aulas, Respostas, Concluidos) using pill buttons
- **Actions:** Optimistic mark-read (individual + all), optimistic delete with rollback on API failure
- **Navigation:** Shared `getNotificationUrl()` utility extracts target URL from notification type and data
- **Empty states:** Two variants — no notifications at all (Bell icon) and no matches for current filter
- **Styling:** Netflix dark theme per design system — unread cards get `border-l-2 border-primary`, read cards are transparent
- **Loading:** Skeleton pulse animation while initial fetch is in progress
- **Accessibility:** Cards use `role="button"` with keyboard support; delete button has aria-label

#### Decisions

1. Extracted `getNotificationUrl()` to `lib/notifications/` as shared utility (story dev note recommendation)
2. "Carregar mais" button hidden when a filter is active (since pagination is server-side but filtering is client-side)
3. `Notification` interface exported from `NotificationCard.tsx` for reuse across components

### Observations

2. **"Carregar mais" hidden during filter:** Button is conditionally rendered only when `filter === 'all'` (line 202), which is the correct approach since pagination is server-side but filtering is client-side. This avoids UX confusion.

Not a blocker since both are compatible, but could be consolidated in a future cleanup.
4. **Keyboard accessibility:** `NotificationCard` has `role="button"`, `tabIndex={0}`, and `onKeyDown` handler for Enter/Space. Good practice.
