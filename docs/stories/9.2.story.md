## Context

### Bug Report

Na página home do membro (`/`), os produtos aparecem corretamente no carrossel "Meus Cursos". Porém, ao clicar em qualquer produto (ProductCard), a URL muda para `/?message=produto-nao-encontrado` e a página recarrega — o usuário nunca chega à página de detalhes do produto.

### Causa Raiz Identificada

A página de detalhes (`(member)/products/[slug]/page.tsx`, linha 83) faz:

```typescript
const { data: product } = await productQuery.single<Product>();
```

O problema é **duplo**:

1. **`.single()` retorna `{ data: null, error }` quando 0 linhas passam o filtro** — mas o `error` é silenciosamente ignorado (não é destructured). Isso impede qualquer diagnóstico.

2. **A RLS policy `products_select_member` depende de uma subquery em `member_access`** — que por sua vez tem suas próprias RLS policies. Se `auth.uid()` não estiver corretamente propagado no contexto RLS do server component, a subquery retorna 0 linhas e o produto fica "invisível".

### Diferença Home vs Detail

| Aspecto | Home Page (funciona) | Detail Page (falha) |
|---------|---------------------|---------------------|
| Query member_access | Explícita: `.eq('profile_id', user.id)` | Implícita via RLS subquery |
| Query products | `.in('id', productIds)` — IDs já validados | `.eq('slug', slug)` — depende 100% de RLS |
| Método | `.select()` (array) | `.single()` (erro se 0 rows) |
| Fallback | Array vazio = sem produtos | null = redirect |

**A home page funciona porque faz uma query explícita em `member_access` primeiro e depois filtra products por IDs conhecidos.** A detail page depende inteiramente da RLS para resolver acesso — se a RLS falha silenciosamente, o produto é tratado como inexistente.

### Mudança Necessária

1. Capturar e logar o `error` do `.single()` para diagnóstico
2. Substituir `.single()` por `.maybeSingle()` — retorna `null` sem erro quando 0 rows (vs `.single()` que retorna erro)
3. Adicionar fallback explícito: se `.maybeSingle()` retorna null para membro, verificar `member_access` diretamente (mesmo padrão da home) e re-tentar a query com `.in('id', [product_id])`

### Decisões Técnicas

- **`.maybeSingle()` em vez de `.single()`** — evita PostgREST error code PGRST116 quando RLS filtra todas as linhas
- **Logging do error** — usar `console.error` com contexto (slug, user.id, error code) para diagnóstico futuro
- **Fallback defensivo** — se RLS falha, query explícita em `member_access` + re-query em products pelo ID (mesma estratégia da home page)
- **Não alterar RLS policies** — o fix é na camada de aplicação para ser resiliente ao comportamento do RLS

### Diferença `.single()` vs `.maybeSingle()`

| Método | 0 rows | 1 row | 2+ rows |
|--------|--------|-------|---------|
| `.single()` | `{ data: null, error: PGRST116 }` | `{ data: row, error: null }` | `{ data: null, error: PGRST116 }` |
| `.maybeSingle()` | `{ data: null, error: null }` | `{ data: row, error: null }` | `{ data: null, error: PGRST116 }` |

### RLS Policies Relevantes

**`products_select_member`** (migration `20260311000008`):
```sql
CREATE POLICY "products_select_member" ON products
    FOR SELECT USING (
        is_published = true
        AND EXISTS (
            SELECT 1 FROM member_access
            WHERE member_access.product_id = products.id
            AND member_access.profile_id = auth.uid()
        )
    );
```

**`member_access`** — tem RLS habilitado com policy para membros verem seus próprios registros.

### Padrão de Fallback (da Home Page)

```typescript
// 1. Query explícita em member_access
const { data: memberAccess } = await supabase
  .from('member_access')
  .select('product_id')
  .eq('profile_id', user.id);

const productIds = (memberAccess || []).map((a) => a.product_id);

// 2. Query products com IDs explícitos
const { data: product } = await supabase
  .from('products')
  .select(`id, title, description, banner_url, slug, modules (...)`)
  .eq('slug', slug)
  .eq('is_published', true)
  .in('id', productIds)
  .maybeSingle<Product>();
```
